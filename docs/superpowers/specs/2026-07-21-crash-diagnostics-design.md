# hua 崩溃诊断设计

日期：2026-07-21

## 背景

2026-07-21 上午，hua 1.0.14 的进程消失，但 Windows Application Error、WER、
可靠性记录和现有 `hua.log` 都没有留下对应证据。当前 `LogLevel=warn` 会过滤启动与
正常退出行，因此无法区分未处理异常、外部强制终止、断电或正常退出。

2026-07-17 的 hua 1.0.6 栈溢出已有系统转储，且当前代码已经通过静态存储规避当时的
大对象栈分配；本设计不把旧问题误判为本次根因。

## 目标与非目标

目标：

- 未处理异常发生时，尽最大努力保存异常摘要和 minidump。
- 下次启动时识别“上次没有走完正常清理”，并以 `WARN` 记录。
- 即使常规日志关闭、日志锁损坏或栈空间不足，也尽量保留崩溃证据。
- 控制转储数量，并清楚说明转储可能包含进程内存片段。

非目标：

- 不尝试从未知的进程损坏状态继续运行。
- 本阶段不自动重启；避免持续崩溃时形成重启循环和反复安装全局钩子。
- 不把强制结束、系统断电或重启一律宣称为“程序崩溃”。

## 方案比较

### 方案 A：只依赖 Windows WER / LocalDumps

实现最少，系统转储质量高，但用户机器未必启用 LocalDumps，且本次事件已经证明可能
完全没有 WER 记录。不能独立满足目标。

### 方案 B：进程内异常记录 + 运行状态标记（采用）

程序安装顶层未处理异常过滤器，写异常摘要和 minidump；启动时创建运行状态标记，只在
正常清理结束后删除。它保持单 EXE、无需管理员权限，并能覆盖“没有异常事件但进程消失”
这一证据缺口。

局限是进程已经损坏时，进程内写转储只能尽力完成。为降低栈溢出场景的失败率，异常过滤
器只复制固定大小数据并唤醒预先创建的专用写入线程；写入线程使用自己的栈，不调用常规
日志器。

### 方案 C：独立守护进程自动拉起和采集

隔离性最好，也能自动重启，但会增加进程、IPC、生命周期和崩溃循环治理，超出当前问题
所需范围。若后续确实需要自动恢复，再单独设计。

## 组件设计

新增独立的 `crash_diag` 模块，避免把异常安全要求混进常规日志实现。

### 初始化与清理

1. 单实例互斥量成功后、配置解析前初始化崩溃诊断。
2. 优先使用 EXE 同级 `crashes` 目录；不可写时回退到
   `%LOCALAPPDATA%\Hua\CrashDumps`。
3. 检查旧运行状态标记，记录其中的版本、PID 和启动时间。
4. 创建本次状态标记，预创建事件和专用转储写入线程，安装顶层异常过滤器。
5. 常规日志初始化后，如发现旧标记，以 `WARN` 写入“上次未正常结束；可能是崩溃、
   强制终止、断电或系统重启”。
6. 仅在消息循环和所有正常清理完成后删除状态标记，再关闭诊断模块。

状态标记使用小型、带版本号的文本格式。写入采用临时文件加原子替换，避免半写内容被
误读。标记不是崩溃证明，只表示正常退出路径没有完成。

### 未处理异常路径

顶层异常过滤器不得调用 `hua_logf`、堆分配、格式化库或持有未知状态的锁。它只做以下
固定工作：

1. 将异常代码、异常地址、线程 ID、`EXCEPTION_RECORD` 和 `CONTEXT` 复制到预分配的
   静态存储。
2. 唤醒专用写入线程，并有限等待写入完成。
3. 返回继续搜索，让 Windows 的默认未处理异常流程仍有机会生成 WER 记录。

写入线程生成同名前缀的一对文件：

- `hua-crash-YYYYMMDD-HHMMSS-mmm-v<version>-p<pid>.txt`
- `hua-crash-YYYYMMDD-HHMMSS-mmm-v<version>-p<pid>.dmp`

文本至少包含版本、进程/线程 ID、异常代码、异常地址、时间、EXE 路径和 dump 写入结果。
minidump 使用 `MiniDumpWriteDump`，包含线程信息和间接引用内存，但不生成完整内存转储，
以控制体积和敏感数据范围。

主线程初始化时调用 `SetThreadStackGuarantee` 预留异常处理栈。其他线程仍由专用写入线程
负责落盘；过滤器等待设置上限，避免转储失败时阻止系统终止进程。

### 保留与隐私

- 最多保留最近 5 对崩溃文件；清理只匹配严格的 `hua-crash-*` 文件名。
- `.dmp` 可能包含窗口标题、路径或其他进程内存片段，只保存在本机，不自动上传。
- 常规 `hua.log` 的开关和保留策略不影响崩溃文件，以免关闭日志后再次失去关键证据。

## 构建与测试

- CMake 为应用链接 Windows SDK 的 `Dbghelp`。
- 为文件命名、状态标记解析、严格保留匹配和旧状态判定编写单元测试。
- 增加隔离的崩溃探针测试：父测试进程启动子进程，子进程安装诊断后故意触发访问冲突；
  父进程验证 `.txt`、`.dmp` 和未清理状态标记均存在。探针禁用系统错误弹窗，避免 CI 卡住。
- 验证正常退出会删除状态标记，强制终止会让下次启动报告未正常结束。
- 运行现有 Debug/Release 测试，并实际启动 Release 版本确认托盘、钩子和单实例行为未回归。
- 按项目约定把版本号从 1.0.14 升到 1.0.15。

## 验收标准

- 故意访问冲突后，10 秒内产生可打开的异常摘要和非空 minidump。
- 故意栈溢出时，专用线程仍能产生异常摘要；若 minidump 失败，摘要必须记录失败码。
- 正常退出不在下次启动产生误报。
- 强制结束后，下次启动出现明确但不武断的 `WARN`。
- 崩溃文件超过 5 对时只删除最旧且严格匹配的文件。
- 所有现有测试和新增测试通过，Release 应用能够正常启动并持续运行。
